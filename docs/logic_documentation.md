## 一、人脸识别考勤逻辑

### 1.1 整体流程图

```
学生拍照 → 图像预处理 → 活体检测 → 人脸检测 → 特征提取 → 特征比对 → 结果判定 → 记录保存
```

### 1.2 详细流程说明

#### 步骤1：图像采集
- **触发方式**：学生点击「拍照考勤」按钮
- **前端操作**：调用摄像头 API 捕获当前帧图像
- **数据传输**：将图像转换为 Base64 编码，通过 HTTP POST 请求发送至 `/api/attendance/checkin`

#### 步骤2：图像预处理
- **后端处理**：
  - 将 Base64 解码为图像格式（JPEG/PNG）
  - 图像尺寸标准化
  - 颜色空间转换（BGR → RGB）

#### 步骤3：活体检测
- **检测目的**：防止照片、视频等欺骗攻击
- **检测方法**：
  - 基于光流法的面部动作检测
  - 3D 深度信息分析（如可用）
  - 纹理分析判断真实性
- **判定结果**：
  - 通过：继续后续人脸比对流程
  - 失败：返回「活体检测失败」错误

#### 步骤4：人脸检测
- **检测工具**：MTCNN（Multi-task Cascaded Convolutional Networks）
- **检测内容**：
  - 定位人脸区域（bounding box）
  - 检测面部关键点（眼睛、鼻子、嘴巴等）
- **处理结果**：
  - 成功：提取人脸区域图像
  - 失败：返回「未检测到人脸」错误

#### 步骤5：人脸特征提取
- **特征提取模型**：FaceNet（InceptionResnetV1）
- **处理流程**：
  - 将人脸图像裁剪为 160×160 像素
  - 通过预训练模型提取 128 维特征向量
  - 特征向量归一化处理

#### 步骤6：特征比对
- **比对数据源**：`student_info` 表中的 `face_feature` 字段
- **比对算法**：余弦相似度计算
- **比对过程**：
  ```
  similarity = dot(feature1, feature2) / (norm(feature1) * norm(feature2))
  ```
- **阈值判定**：
  - 相似度 ≥ 0.8（可配置）：匹配成功
  - 相似度 < 0.8：匹配失败

#### 步骤7：结果判定与记录
- **成功情况**：
  - 查询匹配学生的完整信息
  - 执行情绪分析（见第四章）
  - 保存考勤记录至 `attendance_record` 表
  - 返回学生信息、匹配度、情绪分析结果
- **失败情况**：
  - 记录失败原因（活体检测失败/人脸不匹配/超时）
  - 返回相应错误信息

### 1.3 数据库比对示意图

```
拍照图像 ──→ 特征向量(128维)
                    │
                    ▼
        ┌─────────────────────────┐
        │   student_info 表       │
        │ ┌─────────────────────┐ │
        │ │ student_id          │ │
        │ │ name                │ │
        │ │ class_name          │ │
        │ │ face_feature (JSON) │←─── 遍历比对
        │ │ face_feature_hash   │ │
        │ └─────────────────────┘ │
        └─────────────────────────┘
                    │
                    ▼
            计算余弦相似度
                    │
                    ▼
        最大相似度 ≥ 阈值？
          ┌────────┴────────┐
          │                 │
         是                否
          │                 │
          ▼                 ▼
      匹配成功           匹配失败
```

## 二、学生添加功能逻辑

### 2.1 整体流程图

```
录入学生信息 → 人脸图像采集 → 特征提取 → 数据校验 → 持久化存储
```

### 2.2 详细流程说明

#### 步骤1：学生信息录入
- **录入方式**：
  - 单条添加：填写学号、姓名、班级
  - 批量导入：上传 Excel 文件
- **数据校验**：
  - 学号格式验证（如：20240001）
  - 姓名非空检查
  - 班级名称存在性验证

#### 步骤2：人脸图像采集
- **采集方式**：
  - 摄像头实时拍照
  - 上传本地照片
- **图像要求**：
  - 清晰正面人脸
  - 无遮挡
  - 光线充足

#### 步骤3：人脸检测与特征提取
- **处理流程**：
  - 使用 MTCNN 检测人脸
  - 使用 FaceNet 提取 128 维特征向量
  - 计算特征哈希值（用于去重）

#### 步骤4：数据持久化
- **存储表**：`student_info`
- **存储字段**：
  | 字段名 | 类型 | 说明 |
  |--------|------|------|
  | student_id | VARCHAR | 学生学号（主键） |
  | name | VARCHAR | 学生姓名 |
  | class_name | VARCHAR | 所属班级 |
  | face_feature | JSON | 128维人脸特征向量 |
  | face_feature_hash | VARCHAR | 特征向量哈希值 |
  | feature_version | INT | 特征版本号 |
  | feature_updated_at | DATETIME | 特征更新时间 |
  | status | INT | 状态（1-正常，0-停用） |

#### 步骤5：用户账号创建
- **自动创建**：为新学生创建登录账号
- **默认密码**：`demo_hash_123456`（可修改）
- **存储表**：`user_info`

### 2.3 批量导入流程

```
上传Excel ──→ 解析文件 ──→ 逐条校验 ──→ 人脸采集/导入 ──→ 批量保存
                    │
                    ├─→ 学号重复检查
                    ├─→ 格式验证
                    └─→ 数据完整性检查
```

## 三、合照识别功能逻辑

### 3.1 整体流程图

```
上传合照 → 人脸检测（多人） → 逐个特征提取 → 逐个比对 → 结果汇总 → 情绪分析 → 记录保存
```

### 3.2 详细流程说明

#### 步骤1：照片上传
- **支持格式**：JPG、JPEG、PNG
- **文件大小限制**：≤ 10MB（可配置）

#### 步骤2：多人脸检测
- **检测工具**：MTCNN
- **处理过程**：
  - 遍历图像检测所有人脸
  - 记录每幅人脸的 bounding box
  - 返回人脸数量及位置信息

#### 步骤3：逐个特征提取与比对
- **处理方式**：对每张检测到的人脸依次进行：
  1. 裁剪人脸区域
  2. 提取 128 维特征向量
  3. 与 `student_info` 表进行比对
  4. 记录匹配结果

#### 步骤4：结果汇总
- **匹配结果分类**：
  - 已识别：匹配到学生信息
  - 未识别：未匹配到任何学生
- **存储内容**：
  - 照片基本信息（`group_photo_record` 表）
  - 每个人脸的识别详情（`group_photo_recognition_detail` 表）

### 3.3 数据库表关系

```
group_photo_record           group_photo_recognition_detail
┌──────────────────┐         ┌──────────────────────────────┐
│ photo_id (PK)   │←───────→│ detail_id (PK)               │
│ photo_name      │         │ photo_id (FK)                │
│ photo_path      │         │ student_id (可空)            │
│ activity_name   │         │ class_name                   │
│ activity_time   │         │ status (0/1)                │
│ total_faces     │         │ confidence                   │
│ recognized_count│         │ face_box (JSON)              │
│ unrecognized_count│       │ emotion                      │
└──────────────────┘         │ emotion_confidence          │
                             └──────────────────────────────┘
```

## 四、情绪分析功能逻辑

### 4.1 情绪类型定义

| 情绪类型 | 英文标识 | 说明 |
|----------|----------|------|
| 中性 | neutral | 表情平静，无明显情绪 |
| 开心 | happy | 面带微笑，表情愉悦 |
| 疲惫 | tired | 眼神涣散，表情倦怠 |
| 惊讶 | surprised | 眼睛睁大，表情惊讶 |
| 悲伤 | sad | 嘴角下垂，表情悲伤 |
| 生气 | angry | 眉头紧皱，表情愤怒 |
| 未知 | unknown | 无法识别情绪 |

### 4.2 分析流程

#### 步骤1：人脸区域提取
- 从原始图像中裁剪人脸区域
- 标准化为统一尺寸

#### 步骤2：情绪分类
- **分类模型**：基于深度学习的表情识别模型
- **输出结果**：各情绪类别的概率分布

#### 步骤3：结果判定
- 选择概率最高的情绪类别
- 记录置信度值

### 4.3 情绪记录存储

- **存储表**：`emotion_record`
- **来源类型**：
  - `attendance`：考勤时的情绪分析
  - `group_photo`：合照识别时的情绪分析

## 五、用户认证与授权逻辑

### 5.1 JWT 认证流程

#### 登录流程
```
用户名+密码 ──→ 验证 ──→ 生成Token ──→ 返回 access_token + refresh_token
                                           │
                                           ▼
                              客户端存储到 localStorage
```

#### 访问受保护资源流程
```
请求 ──→ 携带 access_token ──→ 验证Token ──→ 验证通过 ──→ 返回数据
                                      │
                                      └─→ Token过期 ──→ 使用refresh_token刷新
```

### 5.2 角色权限

| 角色 | 权限 |
|------|------|
| admin | 全部权限（学生管理、考勤查看、系统配置） |
| teacher | 考勤查看、合照识别、情绪分析、学生查看 |
| student | 个人考勤、个人情绪查看 |

### 5.3 权限验证流程

```
请求API ──→ 解析JWT ──→ 获取用户角色 ──→ 检查权限 ──→ 执行操作
                                          │
                                          └─→ 权限不足 ──→ 返回403错误
```

## 六、数据存储与管理逻辑

### 6.1 数据库表结构总览

| 表名 | 用途 |
|------|------|
| user_info | 用户账号信息 |
| student_info | 学生基本信息及人脸特征 |
| class_enrollment_history | 学生班级变更历史 |
| attendance_record | 考勤记录 |
| emotion_record | 情绪分析记录 |
| group_photo_record | 合照记录 |
| group_photo_recognition_detail | 合照识别详情 |
| calendar_date | 日期维度表（用于报表统计） |

### 6.2 图像文件存储

- **存储路径**：`data/uploads/`
  - `attendance/`：考勤照片
  - `group_photos/`：合照照片
- **命名规则**：`{日期}/{记录ID}.jpg`
- **数据库存储**：存储相对路径而非二进制数据

### 6.3 数据备份与清理

- **定期备份**：数据库每日备份
- **日志保留**：保留最近 30 天的操作日志
- **旧数据归档**：超过一年的历史数据归档至历史表
